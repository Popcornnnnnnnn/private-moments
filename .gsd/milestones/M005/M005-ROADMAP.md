# M005: AI Media Summaries — Roadmap

## Goal

Deliver a quiet AI summary tool for audio/video moments. Summaries are generated by the Mac server from uploaded media files through local Mac transcription and an external summary API, stored as generated metadata, synced back to iOS, and displayed in a bottom sheet without cluttering the main timeline.

## Slices

### S01 — Contract, Configuration, And Migration Plan

**Goal:** Lock the product/API/data contract before implementation touches schema or sync.

**Outcome:** Shared docs describe AI summary state, provider configuration, privacy boundaries, sync recovery, failure semantics, and output schema.

**Primary requirements:** R014, R015, R016.

**Implementation notes:**

- Choose server env names for provider/model/API key/base URL.
- Define `ai_summaries` and iOS `local_ai_summaries` schema.
- Define server change payloads for summary status/content/deletion.
- Decide whether the first endpoint blocks until completion or returns `202` and completes asynchronously.

### S02 — Server AI Summary Pipeline

**Goal:** Add server-owned AI summary generation without exposing keys or private bodies in logs.

**Outcome:** The server can validate uploaded media availability, run local Mac transcription, call the configured external summary provider, validate structured output, persist result/status, emit server changes, and handle retry/regenerate/delete.

**Primary requirements:** R014, R016.

**Implementation notes:**

- Add a small provider adapter instead of a large AI framework.
- Keep transcript and summary content out of logs.
- Store prompt version, model, provider, input length/hash, status, and error metadata.
- Add focused unit/integration checks for missing media, empty transcript, provider failure, invalid output, and regeneration.

### S03 — iOS Timeline Trigger And Bottom Sheet

**Goal:** Add the user-visible AI summary flow while preserving timeline simplicity.

**Outcome:** Audio/video rows show `Summary ready` only after a ready summary syncs back. The bottom sheet shows ready AI summary content only, with no transcript fallback.

**Primary requirements:** R014, R015.

**Implementation notes:**

- Only show the feature for audio/video media.
- Hide the summary entry until a ready summary exists.
- Keep full summary and transcript content out of the timeline row.
- Persist local summary records and apply server changes cursor-safely.
- Confirm delete summary removes only generated AI metadata.

### S04 — Evaluation, Real-Device UAT, And Documentation

**Goal:** Prove the feature is useful, private, recoverable, and quiet in the real app.

**Outcome:** Builds pass, server/API checks pass, real iPhone verifies audio/video summary flows, summary recovery works after local refresh, logs are inspected for content leakage, and docs are updated.

**Primary requirements:** R001, R002, R003, R014, R015, R016.

**Implementation notes:**

- Build a 10-20 item reference dataset with short/medium/long/no-speech/mixed-language/rambling transcript examples.
- Use code-based checks for structured output validity and no-log content leakage.
- Use human review for summary usefulness and non-invention.
- Verify real iPhone behavior on the paired device with Mac server and external provider configured.

## Completion Bar

- Server schema migration and Prisma generation/build pass.
- Admin build still passes even if Admin only shows minimal diagnostics.
- iOS generic build and focused tests pass.
- Real iPhone can upload and view at least one server-generated audio summary and one server-generated video summary.
- Missing media/empty transcript/provider failure paths are verified.
- Server logs are inspected and do not contain transcript or summary bodies.
- Shared contracts and Chinese human-facing docs are updated.
- M004 media upload success path is completed before this milestone or explicitly recorded as a remaining dependency.

## Implementation Snapshot — 2026-04-30

S01-S03 first implementation has landed:

- Server schema version 7 adds `ai_summaries`; schema version 8 adds v2 document fields to those summary records.
- Server exposes `POST /api/v1/ai/media-summary` and `DELETE /api/v1/ai/media-summary/:summaryId`.
- Server uses a small OpenAI-compatible Chat Completions adapter with structured JSON output, provider/model/prompt metadata, failure states, and privacy-safe logs.
- Sync emits `ai_summary_updated` and `ai_summary_deleted`; iOS stores them in `local_ai_summaries`.
- iOS timeline originally showed summary actions/statuses; later follow-ups superseded this with ready-only `Summary ready` display and no transcript fallback.

Incident follow-up — 2026-05-01:

- Investigation of a recent missing audio summary found that the running server process on port 3210 was an older build and did not enqueue upload-triggered AI jobs for that item at upload time.
- After restarting the current server, manual regeneration failed with `transcription_http_404` because the configured localhost OpenAI-compatible provider did not implement `/audio/transcriptions`.
- `server/src/ai/media-summary.ts` now falls back from transcription 404/405 to Chat Completions audio input by converting the media to temporary WAV with `afconvert`; this path produces a ready structured summary without persisting a transcript hash/length.
- One uploaded audio media item and 4 older uploaded audio items without summary records were backfilled successfully and emitted `ai_summary_updated` server changes through ready state. `inputTranscriptLength` is expected to be `null` for this fallback path.
- Timeline action layout was corrected so the comment button remains right-aligned whether or not `Summary ready` is present.
- Verification passed `server:typecheck`, `server:build`, `git diff --check`, 3210 health, manual missing-audio summary backfill, and `npm run ios:device` with install and launch on the paired iPhone.

- `server/src/ai/media-summary.ts` now detects no-access/no-hearing summary bodies from audio input fallback and throws `audio_input_unusable`.

Remote-only sync diagnostics follow-up — 2026-05-01:

- User reported that Settings > Storage & Diagnostics did not show AI Summaries and audio summaries created on 2026-05-01 still looked missing on iPhone.
- Mac/server checks showed `/api/v1/admin/status` was reachable over localhost and Tailscale and included `aiSummaries`; server DB had `ready=10`, `failed=0`, server change version 194.
- Real iPhone container inspection showed `lastSyncCursor=173`, `local_ai_summaries ready=3 failed=7`, and no pending outbox work.
- Diagnosis: server-side local transcription and summary generation are working, but iOS can remain stale because remote-only `ai_summary_updated` changes are not pulled when no local outbox/upload/download work triggers sync.
- Implemented fix: app foreground, Settings > Storage & Diagnostics refresh, and manual Sync Now originally performed a safe pull even when there was no local outbox/upload/download work. 2026-05-06 follow-up superseded the diagnostics refresh part: Settings > Storage & Diagnostics refresh is now read-only, while app foreground, manual Sync Now, and Pull Server Changes remain the safe pull paths.
- `/api/v1/admin/status` now includes `sync.latestServerChangeVersion`, and iOS Storage & Diagnostics shows `This iPhone cursor`, `Mac change version`, and a behind count to separate server job failures from client cursor staleness.
- Server-change pagination now schedules a follow-up sync when a response reaches the 500-change page size, so remote-only catch-up can continue across large batches.
- Verification copied the paired iPhone container after installing/launching the build: `lastSyncCursor=196`, `local_ai_summaries ready=10 failed=0`, and outbox pending/failed `0`. Server DB showed `MAX(server_changes.version)=196` and `ai_summaries ready=10 failed=0`.
